## 1. 引言
### 1.1. 编制目的
本文档描述了基于 AI 的数智化教学分析评价系统的功能需求和非功能需求，开发人员的软件实现、测试验证与阶段验收工作都将以此为依据。
除特殊说明之外，本文档所包含的需求都是最高优先级需求。

### 1.2. 范围
基于 AI 的数智化教学分析评价系统是一款面向计算机学院任课教师的 Web 端课程学情分析工具，主要提供模板化教学数据上传、学生学情智能分析、知识点掌握热力图展示、学生学习质量评价、AI出题与在线答题、可视化报告生成等功能服务。本系统聚焦单学科单课程的学生学情分析，帮助任课教师直观掌握班级与学生个体的学习状态，辅助教学策略优化，助力学生个性化学习提升。

## 2. 总体描述
### 2.1. 项目前景
当前高校单课程教学中，学生学习数据分散在成绩表、考勤记录、课堂互动、作业文件等多个载体中，任课教师难以快速整合数据、精准掌握每个学生的学习状态与知识薄弱点，学情分析多依赖经验判断，针对性不足。

随着人工智能与大数据技术在教育领域的深度应用，依托数据驱动教学优化已成为课程教学改革的核心方向。本项目旨在构建一套面向任课教师的课程级AI学情分析系统，通过模板化数据上传降低使用门槛，依托智能分析与知识点热力图直观呈现学生学习状态，为教师教学改进、学生学习优化提供科学、量化的支撑依据。

### 2.2. 产品功能
- SF1：系统管理员可以进行系统用户权限管理与操作日志查看。
- SF2：任课教师可以下载标准数据模板，按模板上传本人授课课程的教学数据。
- SF3：任课教师可以查看 AI 智能分析结果，包括学情画像、成绩趋势、知识点掌握热力图、异常学情预警等。
- SF4：系统管理员可以配置学生学习评价指标体系，系统自动完成学生学习质量的量化评价。
- SF5：任课教师可以基于知识点配置AI出题，学生在线完成答题，系统自动更新知识点掌握度。
- SF6：所有授权用户可以查看课程数据看板，生成并导出课程学情分析报告。

### 2.3. 用户特征
| 用户 | 特征 |
|------|------|
| 系统管理员 | 此类用户是系统的运维管理人员，可以通过本系统进行用户管理、评价指标配置、数据模板管理、查看操作日志等。期望系统管理工作便捷高效，配置灵活可控。 |
| 任课教师 | 此类用户是系统核心用户，承担计算机学院课程教学工作，可以通过本系统上传课程教学数据，查看所授课程的学生成绩、考勤、学情分析，查看班级与学生个人学习评价结果。期望能够直观了解班级学习状态与学生知识薄弱点，辅助教学策略优化。 |
| 学生用户 | 此类用户为计算机学院在校修课学生，可以通过本系统查看个人成绩、考勤数据，获取个人学情画像与学习评价结果，参与在线答题。期望能够清晰了解自身学习状态与知识点薄弱点，指导学习优化。 |

### 2.4. 约束
- CON1：系统将发布到 Web 端，仅支持 PC 浏览器访问。
- CON2：项目总开发周期为 4 周，分两个阶段交付与考核。
- CON3：开发团队共 5 人，采用分工协作模式完成开发、测试与文档全流程工作。
- CON4：项目使用 Git 进行版本控制，采用单仓库多模块架构管理代码。
- CON5：所有程序代码必须包含详细中文注释，每个方法需标注功能说明，关键算法需补充逻辑注释。
- CON6：项目需同步交付设计文档、测试报告、汇报材料、过程记录等全套课程材料。

### 2.5. 依赖与假设
- AE1：所有用户都具有 Web 端的基础操作能力，能够熟练使用主流浏览器。
- AE2：任课教师按系统提供的标准模板填写并上传教学数据，数据字段完整有效。
- AE3：服务器与客户端运行环境满足系统软硬件配置要求。
- AE4：评价指标与权重配置符合课程学生学习评价的业务逻辑与实际场景。

#### 3.1.2. 硬件接口
1. 开发所需硬件环境：服务器、客户端 PC、以太网连接。
2. 运行所需硬件环境：基于 B/S 架构，需提供满足配置要求的应用服务器与数据库服务器，以及稳定的网络带宽，保障服务器正常供电。
3. 服务器推荐配置：CPU 4 核及以上，内存 8GB 及以上，可用硬盘空间 50GB 及以上。
4. 客户端推荐配置：普通 PC 机，CPU 双核及以上，内存 4GB 及以上。

#### 3.1.3. 软件接口
1. 采用 MySQL 8.0 及以上版本数据库，服务器必须安装对应版本的 MySQL 数据库作为必要支撑软件。
2. AI 算法服务依赖 Python 3.8 及以上运行环境，提供标准化接口供业务层调用。
3. 前端服务依赖主流现代浏览器渲染，无需客户端安装额外软件。

#### 3.1.4. 通信接口
系统采用 HTTP 和 HTTPS 协议进行前后端通信，内部服务间通过 RESTful API 进行数据交互，接口返回统一 JSON 格式数据。

### 3.2. 功能需求
#### 3.2.1. 新建用户权限
##### 3.2.1.1 特性描述
系统管理员需要新建系统用户并分配对应权限时，一个经过登录验证的管理员账户可以创建新用户账号并配置角色权限。
优先级 = 高

##### 3.2.1.2 刺激响应序列
- 刺激：管理员登录状态下在用户管理界面点击新建按钮
  响应：系统跳转到新建用户权限界面
- 刺激：管理员输入用户账号、姓名、所属角色、所属班级等信息
  响应：系统实时显示输入的用户信息
- 刺激：管理员点击提交
  响应：系统提示是否确认新建用户权限
- 刺激：管理员确认新建
  响应：系统创建相应用户账号与对应权限
- 刺激：管理员取消新建
  响应：系统不新建用户权限，返回用户管理界面

##### 3.2.1.3 相关功能需求
| 编号 | 需求描述 |
|------|----------|
| Rights.New | 系统应该允许管理员用户新建用户权限 |
| Rights.New.UserValid | 系统应该对新建用户权限的管理员用户进行身份验证 |
| Rights.New.UserValid.Logined | 管理员用户必须登录 |
| Rights.New.Confirm | 管理员用户确认新建用户权限 |
| Rights.New.Cancel | 管理员用户取消新建用户权限 |
| Rights.New.Cancel.Manage | 系统返回用户管理界面 |

### 3.3. 非功能需求
#### 3.3.1. 可维护性
- Modifiability1：系统采用模块化分层设计，模块内部组件高内聚，模块之间低耦合，新增功能模块时可快速扩展，降低对现有模块的影响。
- Modifiability2：使用 Git 版本控制系统跟踪和管理代码的更改，保留完整版本迭代记录。
- Modifiability3：所有代码遵循统一开发规范，配备完整中文注释，便于开发人员理解与维护。

#### 3.3.2. 易用性
- Usability1：系统界面布局清晰，操作流程符合高校任课教师与学生的使用习惯，无需专业培训即可上手核心功能。
- Usability2：不同角色用户登录后仅展示对应权限范围内的功能菜单与数据，操作引导与异常提示清晰明确。
- Usability3：确保系统在 Chrome、Edge 等主流现代浏览器上都有一致的用户体验与正常的页面展示效果。
- Usability4：所有列表数据支持筛选、模糊搜索、排序等便捷操作，关键操作执行二次确认提示。

#### 3.3.3. 可靠性
- Reliability1：系统应在各种主流浏览器中均能正常使用，无严重页面兼容问题。
- Reliability2：教学核心数据具备故障后恢复能力，数据导入异常时不影响已有数据，支持操作回滚。
- Reliability3：所有用户关键操作都被记录，操作日志可查询、可追溯、可导出。
- Reliability4：系统具备处理 50 人同时在线访问的并发能力，无明显卡顿与崩溃问题。

#### 3.3.4. 业务规则
- BR1：系统应对教学数据与评价结果进行权限隔离，只有具有相应权限的用户才能访问和修改。
- BR2：教学数据按课程隔离，任课教师仅能查看与管理自己授课课程的数据。
- BR3：评价指标权重配置需通过合法性校验，权重总和必须为 100% 后方可生效。
- BR4：所有教学数据的导入、修改、删除操作均需留痕，支持全流程追溯。

### 3.4. 数据需求
#### 3.4.1. 数据定义
- DR1：用户账号与权限数据永久保存，账号禁用后仍保留历史记录。
- DR2：教学基础数据按课程、学期归档保存，支持跨学期查询对比。
- DR3：AI 分析结果与评价结果永久留存，支持追溯历史版本。
- DR4：系统操作日志保存期限不少于 1 年。
- DR5：题目与答题记录数据按课程永久保存，可用于知识点掌握度持续更新。

#### 3.4.2. 默认数据
- Default1：系统初始化时默认存在一个超级管理员账号，用于初始系统配置。
- Default2：系统内置计算机学院典型课程的仿真模拟数据，包含成绩、考勤、课堂问答、答题记录等，用于开发测试与演示。
- Default3：系统默认提供一套基础学生学习评价指标方案，可直接使用或在此基础上修改。
- Default4：系统预置标准数据模板，包含成绩、考勤、课堂互动三类文件模板。
- Default5：系统预置计算机学院课程知识点体系模板。

#### 3.4.3. 数据格式要求
- Format1：系统所有文本内容采用 UTF-8 编码格式。
- Format2：上传的 Excel 文件仅支持 .xlsx 格式，Txt 文件仅支持英文逗号分隔的 UTF-8 格式。
- Format3：成绩、考勤等数值型数据需符合业务范围校验规则，超出合理范围的数据标记为异常。
- Format4：日期类数据统一采用 YYYY-MM-DD 格式存储与展示。
