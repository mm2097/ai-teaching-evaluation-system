## 1. 引言
### 1.1. 编制目的
本文档描述了基于 AI 的数智化教学分析评价系统的功能需求和非功能需求，开发人员的软件实现、测试验证与阶段验收工作都将以此为依据。
除特殊说明之外，本文档所包含的需求都是最高优先级需求。

### 1.2. 范围
基于 AI 的数智化教学分析评价系统是一款面向高校教学管理场景的 Web 端分析平台，主要提供多源教学数据采集与治理、AI 智能教学分析、多维教学质量评价、可视化报告生成等功能服务。本系统开发应采用成熟稳定的技术，提供全面的教学数据分析能力与便捷的操作模式，力争减少教学评价工作的重复性和繁杂性，提升教学管理效率与决策科学性，保障教学数据安全与信息共享，助力高校教学质量从 “经验驱动” 向 “数据驱动” 转型，提升学校教学信息化管理水平。

## 2. 总体描述
### 2.1. 项目前景
当前高校教学质量评价与教学管理多依赖人工统计、经验判断，教学数据分散在教务系统、成绩表、考勤记录、作业文件等多个异构数据源中，存在数据整合难度大、分析维度单一、评价结果滞后、决策支撑不足等问题，难以实现教学过程的精细化管理与个性化指导。

随着人工智能与大数据技术在教育领域的深度应用，依托数据驱动教学质量提升已成为高校教学改革的核心方向。本项目旨在构建一套基于 AI 的数智化教学分析评价系统，打通多源教学数据链路，通过智能分析与多维评价，实现教学质量从 “经验驱动” 向 “数据驱动” 的转型，为教师教学改进、学生学习优化、教学管理决策提供科学、量化的支撑依据。

### 2.2. 产品功能
- SF1：系统管理员可以进行系统用户权限管理与操作日志查看。
- SF2：数据管理人员可以通过文件上传、数据库直连两种方式接入多源教学数据，并完成数据清洗与标准化治理。
- SF3：教学管理人员与任课教师可以查看 AI 智能分析结果，包括学情画像、成绩趋势、知识点掌握度、异常学情预警等。
- SF4：系统管理员可以配置多维评价指标体系，系统自动完成教师教学质量、学生学习质量、课程建设质量的量化评价。
- SF5：所有授权用户可以查看综合数据看板，生成并导出对应维度的教学分析报告。

### 2.3. 用户特征
| 用户 | 特征 |
|------|------|
| 系统管理员 | 此类用户是系统的运维管理人员，可以通过本系统进行用户管理、评价指标配置、系统参数设置、查看操作日志等。期望系统管理工作便捷高效，配置灵活可控。 |
| 教学管理者 | 此类用户为院系 / 教务处教学管理人员，可以通过本系统查看全院系教学数据总览、教学质量评价结果、学情整体分析、课程质量报告。期望能够便捷获取全局教学分析结论，为管理决策提供支撑。 |
| 任课教师 | 此类用户承担课程教学工作，可以通过本系统查看所授课程的学生成绩、考勤、学情分析，查看个人教学评价结果。期望能够直观了解班级学习状态，辅助教学策略优化。 |
| 学生用户 | 此类用户为在校修课学生，可以通过本系统查看个人成绩、考勤数据，获取个人学情画像与学习评价结果。期望能够清晰了解自身学习状态与薄弱点，指导学习优化。 |

### 2.4. 约束
- CON1：系统将发布到 Web 端，仅支持 PC 浏览器访问。
- CON2：项目总开发周期为 4 周，分两个阶段交付与考核。
- CON3：开发团队共 5 人，采用分工协作模式完成开发、测试与文档全流程工作。
- CON4：项目使用 Git 进行版本控制，采用单仓库多模块架构管理代码。
- CON5：所有程序代码必须包含详细中文注释，每个方法需标注功能说明，关键算法需补充逻辑注释。
- CON6：项目需同步交付设计文档、测试报告、汇报材料、过程记录等全套课程材料。

### 2.5. 依赖与假设
- AE1：所有用户都具有 Web 端的基础操作能力，能够熟练使用主流浏览器。
- AE2：导入系统的教学数据来源真实有效，核心字段完整度满足系统分析要求。
- AE3：服务器与客户端运行环境满足系统软硬件配置要求。
- AE4：评价指标与权重配置符合高校教学评价的业务逻辑与实际场景。

#### 3.1.2. 硬件接口
1. 开发所需硬件环境：服务器、客户端 PC、以太网连接。
2. 运行所需硬件环境：基于 B/S 架构，需提供满足配置要求的应用服务器与数据库服务器，以及稳定的网络带宽，保障服务器正常供电。
3. 服务器推荐配置：CPU 4 核及以上，内存 8GB 及以上，可用硬盘空间 50GB 及以上。
4. 客户端推荐配置：普通 PC 机，CPU 双核及以上，内存 4GB 及以上。

#### 3.1.3. 软件接口
1. 采用 MySQL 8.0 及以上版本数据库，服务器必须安装对应版本的 MySQL 数据库作为必要支撑软件。
2. AI 算法服务依赖 Python 3.8 及以上运行环境，提供标准化接口供业务层调用。
3. 前端服务依赖主流现代浏览器渲染，无需客户端安装额外软件。

#### 3.1.4. 通信接口
系统采用 HTTP 和 HTTPS 协议进行前后端通信，内部服务间通过 RESTful API 进行数据交互，接口返回统一 JSON 格式数据。

### 3.2. 功能需求
#### 3.2.1. 新建用户权限
##### 3.2.1.1 特性描述
系统管理员需要新建系统用户并分配对应权限时，一个经过登录验证的管理员账户可以创建新用户账号并配置角色权限。
优先级 = 高

##### 3.2.1.2 刺激响应序列
- 刺激：管理员登录状态下在用户管理界面点击新建按钮
  响应：系统跳转到新建用户权限界面
- 刺激：管理员输入用户账号、姓名、所属角色、所属院系等信息
  响应：系统实时显示输入的用户信息
- 刺激：管理员点击提交
  响应：系统提示是否确认新建用户权限
- 刺激：管理员确认新建
  响应：系统创建相应用户账号与对应权限
- 刺激：管理员取消新建
  响应：系统不新建用户权限，返回用户管理界面

##### 3.2.1.3 相关功能需求
| 编号 | 需求描述 |
|------|----------|
| Rights.New | 系统应该允许管理员用户新建用户权限 |
| Rights.New.UserValid | 系统应该对新建用户权限的管理员用户进行身份验证 |
| Rights.New.UserValid.Logined | 管理员用户必须登录 |
| Rights.New.Confirm | 管理员用户确认新建用户权限 |
| Rights.New.Cancel | 管理员用户取消新建用户权限 |
| Rights.New.Cancel.Manage | 系统返回用户管理界面 |

#### 3.2.5. 文件方式上传教学数据
##### 3.2.5.1 特性描述
数据管理人员需要将本地 Excel、Txt 格式的教学数据导入系统时，一个经过登录验证的授权用户可以通过文件上传的方式将教学数据存入系统数据库。
优先级 = 高

##### 3.2.5.2 刺激响应序列
- 刺激：授权用户登录状态下在数据管理界面选择文件上传功能
  响应：系统弹出文件选择窗口

  响应：系统解析文件结构，展示字段预览页面
- 刺激：用户完成源字段与系统标准字段的映射配置，点击导入
  响应：系统执行批量数据导入，展示导入进度
- 刺激：导入完成
  响应：系统展示导入结果，包含成功条数、失败条数与异常数据详情

##### 3.2.5.3 相关功能需求

|------|----------|

| Data.FileUpload.UserValid | 系统应该对上传数据的用户进行身份验证 |
| Data.FileUpload.UserValid.Logined | 用户必须登录且具备数据管理权限 |
| Data.FileUpload.Parse | 系统支持解析 Excel 与 Txt 格式的教学数据文件 |
| Data.FileUpload.Mapping | 系统支持源字段与系统标准字段的映射配置 |
| Data.FileUpload.Result | 系统返回导入结果统计与异常数据详情 |

数据管理人员需要对接外部教务数据库获取教学数据时，一个经过登录验证的授权用户可以通过配置数据库连接信息，直接抽取目标教学数据存入系统。
优先级 = 中

##### 3.2.6.2 刺激响应序列
- 刺激：授权用户登录状态下在数据源管理界面点击新增数据库源
  响应：系统弹出数据库连接配置页面
- 刺激：用户输入数据库地址、端口、账号、密码等连接信息，点击测试连接
  响应：系统验证连接有效性，返回连接测试结果
- 刺激：连接成功后，用户选择目标数据表与字段，确认接入
  响应：系统执行数据抽取，将目标数据同步至系统数据库
- 刺激：抽取完成
  响应：系统展示数据接入结果与数据统计

##### 3.2.6.3 相关功能需求
| 编号 | 需求描述 |

### 3.3. 非功能需求
#### 3.3.1. 可维护性
- Modifiability1：系统采用模块化分层设计，模块内部组件高内聚，模块之间低耦合，新增功能模块时可快速扩展，降低对现有模块的影响。
- Modifiability2：使用 Git 版本控制系统跟踪和管理代码的更改，保留完整版本迭代记录。
- Modifiability3：所有代码遵循统一开发规范，配备完整中文注释，便于开发人员理解与维护。

#### 3.3.2. 易用性
- Usability1：系统界面布局清晰，操作流程符合高校教学管理人员与师生的使用习惯，无需专业培训即可上手核心功能。
- Usability2：不同角色用户登录后仅展示对应权限范围内的功能菜单与数据，操作引导与异常提示清晰明确。
- Usability3：确保系统在 Chrome、Edge 等主流现代浏览器上都有一致的用户体验与正常的页面展示效果。
- Usability4：所有列表数据支持筛选、搜索、排序等便捷操作，关键操作执行二次确认提示。

#### 3.3.3. 可靠性
- Reliability1：系统应在各种主流浏览器中均能正常使用，无严重页面兼容问题。
- Reliability2：教学核心数据具备故障后恢复能力，数据导入异常时不影响已有数据，支持操作回滚。
- Reliability3：所有用户关键操作都被记录，操作日志可查询、可追溯、可导出。
- Reliability4：系统具备处理 50 人同时在线访问的并发能力，无明显卡顿与崩溃问题。

#### 3.3.4. 业务规则
- BR1：系统应对教学数据与评价结果进行权限隔离，只有具有相应权限的用户才能访问和修改。
- BR2：评价指标权重配置需通过合法性校验，权重总和必须为 100% 后方可生效。
- BR3：所有教学数据的导入、修改、删除操作均需留痕，支持全流程追溯。

### 3.4. 数据需求
#### 3.4.1. 数据定义
- DR1：用户账号与权限数据永久保存，账号禁用后仍保留历史记录。
- DR2：教学基础数据按学期归档保存，支持跨学期查询对比。
- DR3：AI 分析结果与评价结果永久留存，支持追溯历史版本。
- DR4：系统操作日志保存期限不少于 1 年。

#### 3.4.2. 默认数据
- Default1：系统初始化时默认存在一个超级管理员账号，用于初始系统配置。
- Default2：教学数据列表初始为空，需通过导入或接入方式录入。
- Default3：系统默认提供一套基础教学评价指标方案，可直接使用或在此基础上修改。
- Default4：系统预置学期、院系、专业等基础数据字典模板。

#### 3.4.3. 数据格式要求
- Format1：系统所有文本内容采用 UTF-8 编码格式。
- Format2：上传的 Excel 文件支持 .xlsx、.xls 格式，Txt 文件支持逗号、制表符分隔格式。
- Format3：成绩、考勤等数值型数据需符合业务范围校验规则，超出合理范围的数据标记为异常。
- Format4：日期类数据统一采用 YYYY-MM-DD 格式存储与展示。
