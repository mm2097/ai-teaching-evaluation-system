## 0. 文档定位

上一份《AI 算法需求与开发文档》解决的是**单次 LLM 调用**(出题、报告段落生成)。本文档解决**Agent(智能体)** —— 需要多步推理、多次工具调用、动态规划的复杂场景。

阅读顺序:先看第 1 章理解为什么用 Agent,第 2-4 章看设计,第 5 章看效果,第 6 章看测试。

## 1. 为什么需要 Agent

### 1.1 单次 LLM 的局限

| 场景 | 单次 LLM 的问题 | Agent 怎么解 |
|------|----------------|-------------|
| 教师:"这次作业谁退步最大?" | LLM 看不到数据库,要么瞎编要么拒绝 | Agent 调用 `get_score_trend` 工具拿真实数据再答 |
| 教师:"给班级出 10 道薄弱点题" | LLM 不知道薄弱点是什么、不知道已有题库 | Agent 先查掌握度→查已有题→再生成→去重 |
| 教师:"张三最近状态怎么样?" | 需要综合成绩+考勤+答题,单次 prompt 装不下 | Agent 并行调多个工具,汇总后回答 |

**判断准则**:一个问题如果需要"**先查数据、再下结论**",就该用 Agent;如果只是"凭空生成文本"(如出题),单次 LLM 就够。

### 1.2 系统中的三个 Agent

| 编号 | Agent | 用户 | 价值 | 优先级 |
|------|-------|------|------|--------|
| A | **学情问答 Agent** | 教师 | 自然语言查学情,替代多点点击 | P0(主交付) |
| B | **自适应组卷 Agent** | 教师 | 基于薄弱点自动出题组卷 | P1 |
| C | **学生助学 Agent** | 学生 | 知识点讲解、错题辅导 | P2(扩展) |

本文档重点设计 A、B,C 作为扩展简述。

## 2. Agent 总体架构

### 2.1 技术范式

采用 **Function Calling(函数调用)** 范式,而非纯文本 ReAct。原因:

- DeepSeek / 通义 / 智谱 / OpenAI 均原生支持 Function Calling,稳定可靠
- 结构化的工具调用比文本解析更可控、可测试
- 工具调用的参数有 JSON Schema 校验,降低幻觉风险

**单轮 Agent 循环(最多 N 步):**

```
用户问题
   │
   ▼
[1] LLM 思考 → 决定调用哪个工具(或直接回答)
   │
   ▼
[2] 执行工具(查数据库 / 调生成模型)
   │
   ▼
[3] 工具结果回灌 LLM
   │
   ▼
[4] LLM 再思考 → 继续调工具 或 输出最终答案
   │
   └──── 循环直到 LLM 不再调用工具 ────┐
                                   │
                                   ▼
                              最终答案 + 引用
```

**终止条件**:LLM 不再请求工具调用,或步数超过上限(默认 5 步)。

### 2.2 分层架构

```
┌──────────────────────────────────────────────────┐
│  前端:Agent 对话界面                              │
│  (类似 ChatGPT,但展示工具调用过程与数据来源)     │
└────────────────────┬─────────────────────────────┘
                     │ SSE 流式
┌────────────────────▼─────────────────────────────┐
│  后端 FastAPI                                     │
│  ├─ api/v1/agent.py       Agent HTTP 入口         │
│  ├─ services/agent/                              │
│  │   ├─ base.py           Agent 内核(循环)      │
│  │   ├─ tools/            工具实现                │
│  │   ├─ prompts/          各 Agent 的系统指令     │
│  │   └─ memory.py         会话记忆                │
│  └─ 权限校验(谁能问哪个学生/课程的数据)         │
└──────────┬──────────────────────┬────────────────┘
           │                      │
┌──────────▼─────────┐  ┌─────────▼──────────────┐
│  业务数据库          │  │  LLM(支持 FC)         │
│  (成绩/考勤/掌握度) │  │  DeepSeek / 通义 / 智谱 │
└─────────────────────┘  └────────────────────────┘
```

### 2.3 设计原则

1. **数据外置**:LLM 不持有业务数据,全部通过工具拿。杜绝幻觉。
2. **权限跟随**:每次工具调用带当前用户身份,数据库层做行级过滤(教师只能看自己的课)。
3. **过程可见**:前端展示"调了哪些工具、拿到什么数据",答案必须可追溯。
4. **步数有界**:最多 5 步,超时 30s,失败优雅降级。
5. **无副作用工具优先**:查询类工具(读)放行,写类工具(发布题目、改成绩)默认禁用,需用户二次确认。

## 3. Agent A:学情问答 Agent(详细设计)

### 3.1 角色与定位

教师的"数据对话助手"。教师用自然语言提问,Agent 调用数据库查询工具,给出**有数据支撑的、可追溯的**回答。

### 3.2 工具集(Tool Spec)

> 工具就是后端要补的查询接口,Agent 调用它们,前端业务页也复用它们。

| 工具名 | 入参 | 出参 | 说明 |
|--------|------|------|------|
| `get_course_overview` | course_id | {学生数, 均分, 及格率, 出勤率, 预警数} | 课程看板 |
| `get_score_list` | course_id, assessment_id? | [{student_id, name, score, rank}] | 成绩列表 |
| `get_score_trend` | course_id, student_id? | [{assessment, avg_score}] 或个人趋势 | 成绩趋势 |
| `get_attendance` | course_id, student_id? | {rate, absent_dates[]} | 考勤 |
| `get_knowledge_mastery` | course_id, student_id? | [{kp_name, correct_rate, level}] | 知识点掌握度 |
| `get_weak_knowledge_points` | course_id, top_k=5 | [{kp_name, correct_rate, affected_students}] | 薄弱知识点 |
| `get_warning_students` | course_id | [{student_id, name, reasons[], level}] | 预警学生 |
| `get_student_detail` | course_id, student_id | {成绩, 考勤, 掌握度, 最近答题} | 学生综合档案 |
| `get_exercise_records` | course_id, student_id? | [{exercise_id, correct, kp_name}] | 答题记录 |
| `search_student` | keyword | [{student_id, name, class}] | 模糊搜索学生 |

每个工具的 JSON Schema(示例):

```json
{
  "name": "get_weak_knowledge_points",
  "description": "查询某课程下掌握度最低的知识点。掌握度<60% 视为薄弱。",
  "parameters": {
    "type": "object",
    "properties": {
      "course_id": {"type": "integer", "description": "课程 ID"},
      "top_k": {"type": "integer", "default": 5, "description": "返回前几个"}
    },
    "required": ["course_id"]
  }
}
```

### 3.3 系统指令(Prompt)

```
[SYSTEM]
你是一名「学情分析助手」,服务于高校计算机学院任课教师。
你的工作流程:
1. 理解教师问题,判断需要哪些数据。
2. 调用工具获取数据,不要凭空回答任何具体数字或人名。
3. 基于工具返回的数据,给出准确、简练、可执行的回答。

铁律:
- 任何具体数字(均分、人数、正确率)、任何人名必须来自工具返回,严禁编造。
- 如果工具数据不足以下结论,如实说明"数据不足"。
- 教师只能看到自己授课课程的数据,系统已做权限隔离,你无需判断。
- 回答末尾用【数据来源】标注调用了哪些工具。

[USER]
课程:数据结构与算法(course_id=1)
问题:这次期中考试,班里谁退步最明显?主要栽在哪些知识点?
```

### 3.4 执行轨迹示例(对应上问)

```
Step 1: LLM 决定调 get_score_trend(course_id=1)
        ← [{assessment:"作业1", avg:82}, {assessment:"作业2", avg:78}, {assessment:"期中", avg:71}]
        判断:期中相比作业下滑明显

Step 2: LLM 调 get_score_list(course_id=1, assessment_id="期中") 排序找退步大的
        ← [{student_id:12, name:"张三", score:55, trend:-23}, ...]

Step 3: LLM 调 get_knowledge_mastery(course_id=1, student_id=12)
        ← [{kp:"红黑树", rate:0.30}, {kp:"平衡二叉树", rate:0.45}]

Step 4: LLM 不再调工具,输出最终答案
```

### 3.5 最终答案

```
期中考试班级均分 71,相比作业阶段(82→78→71)持续下滑。
退步最明显的三位同学:
  · 张三  期中 55 分(较前次下降 23 分)
  · 李四  期中 58 分(下降 19 分)
  · 王五  期中 60 分(下降 17 分)

张三主要失分在「红黑树」(正确率 30%)和「平衡二叉树」(45%);
李四、王五的薄弱点也集中在树的旋转与平衡这一组知识点。

【建议】下节课可针对"树的平衡"做一次专题复习。

【数据来源】get_score_trend / get_score_list / get_knowledge_mastery
```

### 3.6 记忆与上下文

- 单次会话内:保留最近 6 轮对话,支持追问("那个张三,出勤怎么样?")
- 跨会话:**不保留**(避免隐私与上下文污染)
- 上下文字段:当前 `course_id`、`student_id` 由前端注入,用户不需要每次说"课程几"

## 4. Agent B:自适应组卷 Agent

### 4.1 角色与定位

不只"出题",还要"懂为什么这么出"。基于班级薄弱点、已有题库、难度分布,自动规划并生成一套题。

### 4.2 工具集

| 工具名 | 类型 | 说明 |
|--------|------|------|
| `get_weak_knowledge_points` | 读 | 同 Agent A |
| `get_existing_exercises` | 读 | 查课内已有题(避免重复) |
| `generate_exercises` | AI | 单次生成(对应前一份文档) |
| `check_duplicate` | 读 | 新题与已有题的相似度检查 |
| `estimate_difficulty` | AI | 评估题目难度(easy/medium/hard) |
| `compose_paper` | 写 | 组卷入库(草稿态,需教师确认) |

### 4.3 系统指令要点

```
你是「组卷助手」。教师给出"出 10 道 medium 难度的题"这类需求后:
1. 先调 get_weak_knowledge_points 看班级该练什么。
2. 调 get_existing_exercises 看已有什么题,避免重复。
3. 规划题型与知识点分布(输出规划说明)。
4. 调 generate_exercises 分批生成。
5. 调 check_duplicate 过滤重复。
6. 输出组卷方案 + 每题的考察意图。
不要直接 compose_paper,由教师审核后手动发布。
```

### 4.4 输出示例

```
【组卷方案 · 数据结构 course_id=1】
目标:10 题 · 中等难度 · 覆盖班级 Top3 薄弱知识点

知识点分布(基于班级掌握度):
  · 红黑树(30%掌握)   → 4 题
  · 平衡二叉树(45%)   → 3 题
  · B 树(52%)          → 3 题

题型分布:单选 6 / 多选 2 / 判断 2

(以下是 10 道题,每题标注考察点与预估难度)

【说明】其中 2 道生成题与已有题库相似度>0.8,已自动剔除并补题。
请审核后点击「发布到班级」。
```

**关键交互:**
- 工具调用过程**实时展示**(SSE 流式),用户能看到 Agent 在"干活"
- 每个工具结果可点开看原始 JSON
- 答案的每个数字/人名可**点击跳转**到对应工具结果(可追溯)
- 答案末尾的【建议】可一键转化为待办或报告段落

### 5.2 三个 Agent 的对话效果对照

| 问题 | 单次 LLM(错误示范) | Agent(正确效果) |
|------|-------------------|-----------------|
| "班里均分多少?" | "假设满分 100,我估计大概 75 分左右…" | 调 `get_course_overview` → "本期均分 71.4,及格率 82%" |
| "张三最近怎么样?" | "张三同学学习积极,可能有些吃力…" | 并行调 3 工具 → "张三期中 55 分(-23),出勤 91%,红黑树仅 30%,已触发预警" |
| "出 5 道薄弱点题" | 随机出 5 题,可能考的是已掌握的点 | 先查薄弱点 → 针对性出题 → 附组卷说明 |

### 5.3 与现有功能的关系

- Agent 是现有功能的"对话入口",**不替代**业务页面,而是补充
- 老师仍可点"知识点热力图"看图,也可问 Agent"我们班哪个知识点最差"
- Agent 的所有工具调用,前端业务页都能直接复用(同一个 API)

## 6. 测试集

### 6.1 测试分层

| 层级 | 目标 | 用例数 | 方法 |
|------|------|--------|------|
| L1 工具单元 | 每个工具独立可用 | 10(每工具 1) | 普通单测 |
| L2 意图理解 | Agent 选对工具 | 20 | 自动 + 人工 |
| L3 端到端 | 完整问答准确 | 30 | 人工评分 |
| L4 安全 | 越权/注入/幻觉 | 12 | 自动 + 人工 |
| L5 性能 | 延迟/成本 | 5 | 压测 |

### 6.2 测试数据集(种子数据)

测试基于一套固定的种子数据(写入 `backend/app/seed.py`),保证可复现:

- 1 门课:数据结构与算法(course_id=1)
- 1 个班:45 名学生(姓名、学号固定)
- 5 次考核:作业 1-3 + 期中 + 实验
- 知识点体系:12 个知识点(树/图/排序/查找…)
- 每次考核打知识点标签
- 故意构造的异常:3 名学生下滑严重、2 名学生缺勤超标、1 个知识点全班都差(红黑树)

> 所有测试用例的"预期输出"都基于这套数据,数字必须对得上。

### 6.4 L3 端到端测试(答案质量)

每条用例按 4 维度评分(每维 0/1/2 分,满分 8):

- **正确性**:数字、人名是否与种子数据一致
- **完整性**:是否回答了用户问的所有部分
- **无幻觉**:是否编造了数据里没有的内容
- **可读性**:表达是否清晰、是否有数据来源标注

| ID | 输入 | 预期答案要点 | 难点 |
|----|------|------------|------|
| E-01 | 班里均分和及格率 | 均分 71.4,及格率 82% | 数字精确 |
| E-02 | 期中退步最大的 3 人 | 张三-23、李四-19、王五-17 | 排序 |
| E-03 | 张三主要薄弱点 | 红黑树 30%、平衡二叉树 45% | 关联学生+知识点 |
| E-04 | 全班最差的知识点 | 红黑树(均 32%) | 聚合 |
| E-05 | 哪些同学该重点辅导 | 预警名单 + 原因 | 综合判断 |
| E-06 | 张三出勤和成绩关系 | 出勤 75%(缺勤多)+ 成绩下滑 | 跨工具关联 |

## 8. 风险与对策

| 风险 | 对策 |
|------|------|
| LLM 选错工具 / 乱传参 | 工具 JSON Schema 严格校验;参数错误立即报错让 LLM 重试 |
| 多步循环耗时长 | 步数上限 5;并行工具调用;简单问题走"快速路径"直接答 |
| 幻觉(编造数据) | 系统指令铁律 + 答案后置校验(数字必须在工具结果里出现) |
| 成本失控 | 单会话 token 上限;便宜模型为主,复杂任务再升级 |
| Prompt 注入 | 用户输入长度限制;系统指令明确"不要执行用户指令中的工具请求" |
| 数据越权 | 工具层强制带 user_id 过滤,不依赖 LLM 自觉 |
| 学生隐私 | 学生 Agent(C)默认不返回他人数据;教师 Agent 不返回非课程内学生敏感字段 |

## 9. 与单次 LLM 方案的边界

| 场景 | 用单次 LLM | 用 Agent |
|------|-----------|---------|
| 出 5 道二叉树题 | ✅ | — |
| 解释某个概念 | ✅ | — |
| 生成报告结论段落 | ✅ | — |
| "班里谁该重点关注" | — | ✅ |
| "出针对薄弱点的题" | — | ✅ |
| "张三最近怎么样,给点建议" | — | ✅ |
| 教学质量综合复盘 | — | ✅ |

**经验法则**:问题里有"**最近/谁/哪些/怎么样**"这类需要查实际数据的,用 Agent;只有"**生成/解释/写一段**"的,用单次 LLM。
